Where We Are

The landscape is changing fast. The engineers who learn to orchestrate, manage context, and enforce quality will pull ahead. Standing still means falling behind.

The Real Constraint

AI Is a Mirror for Unclear Thinking

When the output is messy, the input usually was too.

Ambiguous Words

A word that meant one thing to you and something completely different to the model. "Service" could mean a backend class, a microservice, or a SaaS product.

Implied Constraints

Some requirement you thought was obvious but never stated. "Of course it should use our existing auth flow." The model doesn't know that unless you say it.

Unmade Decisions

A design choice you hadn't actually settled yet but expected the AI to make correctly on your behalf. If you don't know the answer, neither will the model.

The Black Box

Every AI instance is born with zero context about your project, your team, or your conventions. It's in the codebase but it has no memory of yesterday.

The unlock with agentic coding is not what the LLMs do. It's how you manage and organize information and context.

The Thesis

Context is Everything

Every AI session starts from zero. Your codebase is a black box unless you engineer otherwise.

The Ceiling

The limit on AI results isn't model intelligence. It's what the model sees before it acts. Better context beats a better model every time.

The Compound Effect

Every hour you invest in structuring your documentation, CLAUDE.md files, and skills pays back across every future session. Context compounds. It never forgets.

The Window

Teams that invest now will be months ahead by end of year. This isn't optional optimization. It's the difference between 2x and 10x.

The rest of this session is about how to do this: engineer your context so Claude delivers predictable, high-quality results every time.

Core Concept

The Agentic Loop

Every task follows the same cycle. Understanding this loop changes how you prompt and steer Claude.

Your Prompt
→
Gather Context
→
Take Action
→
Verify Results
→
Repeat

Gather Context

Searches files, reads code, runs commands to understand the current state. Every file it reads eats context window. This is why scoping matters.

Take Action

Edits files, runs shell commands, creates branches. Claude picks the right tool at each step based on what it learned from the last. You can interrupt any time.

Verify Results

Runs tests, checks build output, validates behavior. If you give Claude verification criteria up front, output quality jumps dramatically.

The single highest-leverage thing you can do: give Claude a way to verify its own work.

Tests, expected output, linting, type checks. If Claude can check itself, you become a reviewer, not a debugger.

Most Underused Features

Resume & Rewind

Your sessions are persistent and reversible. Use this to your advantage.

Resume
> /resume
# Pick from recent sessions
  • Pick up exactly where you left off
  • Fork sessions: open a new terminal, run /resume, and select the same session. Now you have two instances with the same context.
  • Or select an older session to pick up previous work
Rewind
# Double-tap Escape or:
> /rewind
# Restore to any checkpoint
  • Undo mistakes: revert to the state before the agent messed something up
  • Context preservation: build up great context, fix a small bug, rewind, fix another small bug, rewind. Same rich context every time.
Extend Claude Code

Skills

Skills are reusable workflows stored in your repo. Claude loads them on demand when they're relevant, or you invoke them directly with a slash command.

What They Are

A SKILL.md file inside .claude/skills/ that defines a prompt, a workflow, or domain knowledge Claude should use for specific tasks.

  • Checked into git, shared with your team
  • Loaded on demand, not every session (unlike CLAUDE.md)
  • Can be invoked manually: /bug-fixer, /code-mapper
  • Or auto-applied when Claude detects relevance

What They Unlock

Consistent, repeatable workflows that any team member can trigger with a single command.

  • Team conventions as code. Not in a wiki nobody reads. In the repo, enforced by the tool.
  • Onboarding accelerator. New engineer runs /fix-issue 1234 and follows a proven workflow.
  • Quality guardrails. The skill defines the steps: research, plan, verify. The engineer just invokes it.
# .claude/skills/bug-fixer/SKILL.md
---
name: bug-fixer
description: Investigate bugs with root cause analysis before fixing
---
Analyze the bug: $ARGUMENTS
1. Work in read-only mode first. Understand the problem.
2. Find evidence: specific files, line numbers, log output.
3. Root cause analysis. Narrow to the most likely cause.
4. Propose a clean fix. Pause for approval before editing.
Scaling Your Sessions

Sub-agents & Context Management

Sub-agents

Claude can spawn separate workers that explore in their own context window and report back summaries.

  • Research without cluttering your main session
  • Code review with a fresh perspective
  • Parallel investigation across different areas
"Please investigate the structure
of the codebase. Spin up a sub-agent
for each major service or feature."

Context Window

Your most important resource. It holds everything: conversation, files read, command output. Performance degrades as it fills.

  • /clear between unrelated tasks
  • /compact to summarize what matters and free space. You can even add a focus: /compact focus on the API changes
  • Use sub-agents for heavy exploration
  • After 2 failed corrections: /clear and start fresh with a better prompt
The Router Pattern

CLAUDE.md as a Router

A markdown file Claude reads at the start of every session. Purpose, structure, and pointers to where to go next.

# Frontend Components

## Purpose
Component organization following feature-first layout
with progressive hoisting as code becomes reusable.

## Directories at This Level
| Directory | Purpose | Docs |
|------------|--------------------------------|-----------------------------|
| ui/ | shadcn/ui primitives | → ./ui/CLAUDE.md |
| common/ | Shared composites by concern | → ./common/CLAUDE.md |
| pages/ | Feature pages by domain | → ./pages/CLAUDE.md |
| layout/ | Layout, breadcrumbs, headers | (no guide yet) |

## Quick Reference
New feature? Start in pages/<feature>/
Reusable across features? Move to common/<group>/
Design-system worthy? Promote to ui/

Each subdirectory CLAUDE.md follows the same pattern: brief purpose, pointers to where to go, quick reference for common tasks.

The Router Pattern

The Trail Marker Analogy

Without CLAUDE.md

Claude is hiking through the woods, hoping to stumble on the right landmarks.

  • Greps files, reads directories, guesses
  • Wastes context on dead ends
  • Misses hidden configuration
  • May never find the critical file

Lost in the dark. Exploring by touch.

With CLAUDE.md Files

Trail markers at every fork in the path. Claude walks in, sees where to go.

  • Knows the boundaries and architecture upfront
  • Follows markers to the right subdirectory
  • No dead ends, no wasted context
  • Junior engineer + good CLAUDE.md = senior-level output

Guided, efficient, precise.

The Router Pattern

Hierarchical Loading

CLAUDE.md files form a tree. Root loads always. Children load when Claude works in those directories. Progressive disclosure.

Root
CLAUDE.md — Project-wide: code style, build commands, architecture overview, team conventions
Services
apps/api/CLAUDE.md — API patterns, route conventions, DB access rules, auth flow
Feature
apps/api/src/services/scoring/CLAUDE.md — Scoring engine internals, rule evaluation, test patterns
Frontend
apps/web/src/components/CLAUDE.md — Component patterns, hoisting rules, naming conventions

T-shaped context: broad at the top, deep where Claude is working.

Claude doesn't load everything upfront. It follows the trail. Each layer adds only what's needed for that area of the codebase.

The Router Pattern

What Goes In, What Stays Out

A CLAUDE.md is only valuable if it's concise. For every line, ask: would removing this cause Claude to make mistakes?

Include
  • Build commands Claude can't guess
  • Code style rules that differ from defaults
  • Architectural decisions specific to your project
  • Common gotchas and non-obvious behaviors
  • Pointers to important subdirectories
Exclude
  • Anything Claude can figure out from code
  • Standard language conventions
  • Long tutorials or API docs (link instead)
  • Information that changes frequently
  • Self-evident things like "write clean code"

Bloated CLAUDE.md files cause Claude to ignore your instructions and pollutes your context window.

Treat it like code: review it when things go wrong, prune regularly, and test whether Claude's behavior actually shifts.

Skills & Prompts

Build the Factory, Not the Widget

Instead of copy-pasted prompts, learn how to think about building them. They'll change with every project, model, and team member.

The Universal Pattern

Every effective prompt follows the same skeleton:

Familiarize
→
Research
→
Plan
→
Verify
  • Familiarize: understand the codebase and its patterns
  • Research: dig into docs for unfamiliar technologies
  • Plan: define exactly what changes to make and where
  • Verify: confirm the strategy works with existing conventions

Store & Share

Keep skills in .claude/skills/ in your repo. They're shareable, version-controlled, and Claude loads them on demand.

.claude/skills/
  bug-fixer/
    SKILL.md
  codebase-mapper/
    SKILL.md
  implementation-planner/
    SKILL.md

These prompts will change constantly. That's the point. The skill is knowing how to build them.

Skill Example

Implementation Planner

Before writing any code, produce a complete plan with specific file-level changes. Think of it as a blueprint.

1
Explore the codebase. Search for related logic, constants, schemas, and patterns. Be exhaustive. Leave no dependency unaccounted for.
2
Consider multiple approaches. Explore different design strategies before committing. Document tradeoffs. Prefer de-duplication, strong typing, and single sources of truth.
3
Write a file-by-file change list. For every file created, modified, or deleted: exact path, before-and-after snippets, and a clear explanation of why.
4
Build a granular task list. Numbered, checkboxed, with sub-tasks. Each task is independently actionable. This is what the executor will follow.

The plan must be self-contained. Another engineer (or another Claude session) should be able to execute it without asking questions.

Skill Example

Bug Fixer

Read-only investigation first. Propose changes only after understanding the root cause. Never jump straight to a fix.

1
Understand the problem. Read logs and error output. Identify what the system is doing and where the failure occurs. Explain the behavior in clear terms.
2
Find evidence. Point to specific lines in logs. Reference exact file paths and line numbers. Use search tools to explore the codebase for related function names and error strings.
3
Root cause analysis. Think through multiple possible causes. Narrow down to the most likely. Describe why the current logic breaks and under what conditions.
4
Propose a clean fix. Present the simplest solution that aligns with existing patterns. No band-aids. Include exact file paths, line numbers, and before-and-after code. Then pause for approval.

Key principle: work in read-only mode first. Don't touch files until you understand the problem and the human approves.

Skill Example

Codebase Mapper

Create a comprehensive, navigable map of a codebase so any engineer (or any Claude session) can immediately understand the project.

1
Explore the foundation. Map the root directory, major directories, config files, build systems, and test frameworks. Get the big picture first.
2
Deep structure analysis. Spin up a sub-agent for each major service or feature. Identify naming conventions, file patterns, architecture patterns (MVC, services, repositories). Sample multiple files of each type to understand the conventions.
3
Catalog every component. For each major module: location, purpose, key files, dependencies, and what consumes it. Be exhaustive. Use real file paths, not generalizations.
4
Build the quick-reference table. "Looking for API endpoints? src/routes/. Business logic? src/services/." Make it scannable. Think like a navigator.

The final map should let anyone immediately understand the architecture, locate any code, and know where to make changes.

Recap

Key Takeaways

1
AI is a mirror for unclear thinking. Messy output means messy input. Write thorough prompts, state constraints explicitly, make your decisions before asking AI to implement them.
2
Context is your most important resource. Manage it aggressively. Use /clear between tasks, sub-agents for exploration, and /rewind when things go sideways.
3
CLAUDE.md files are trail markers. Build a hierarchical router that gives Claude the right context at the right time. This is the single biggest multiplier.

Engineer your workflow. PRDs define the what. Implementation plans define the how. Documentation compounds over time. Don't vibe code production software.
